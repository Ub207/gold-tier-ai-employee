### Problem



When using the Human-in-the-loop (HITL) feature, I can approve or reject a

pending tool call via `RunState.reject()`. However, there is currently no way to

tell the model **why** the call was rejected.



The only workaround is to set a global `RunConfig.tool_error_formatter` callback

at run-configuration time, but that callback receives no information about which

specific call was rejected or what the user actually said, so I cannot pass a

meaningful, per-call reason to the model.



This means the model always receives a generic message like

`"Tool execution was not approved."` — even when the user typed an explicit

reason such as:



- "Location access was denied because the user is in a restricted area."

- "This action requires manager approval; please escalate."

- "The requested file does not exist on this machine."



The model then has no context to adjust its next response or ask a better

follow-up question.



### Desired behavior



`RunState.reject()` should accept an optional `rejection_message` parameter so

that the specific reason can be forwarded to the model:



```python

state = result.to_state()

for interruption in result.interruptions:

    reason = ask_user_why_they_rejected(interruption)   # e.g. from a UI text box

    state.reject(

        interruption,

        rejection_message=reason,   # ← pass the actual reason

    )

result = await Runner.run(agent, state)

```



## Why this matters



In real HITL flows, the human operator often knows exactly why they are declining an action and wants the agent to understand that context so it can:



- Avoid repeating the same request

- Suggest an alternative approach

- Give the end-user an accurate explanation



Without this feature, all of that context is silently discarded at the rejection step.
