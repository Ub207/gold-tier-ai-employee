Hey Ubaid ur rehman,



A lot of people still think the big change in software

engineering is that AI can write code now.



I do not think that is the real shift.



The real shift is that writing code is no longer the bottleneck.



Understanding it is.



A model can generate more code in an hour than most people can

properly review in a day.



And that sounds great until you are staring at a massive diff,

trying to figure out what actually changed, why it was done that

way, and whether any of it will be a problem three weeks from

now.



That is the part people keep missing…



Speed is not the hard part anymore.



The hard part is keeping the codebase understandable after all

that speed gets introduced.



A few years ago, if you wrote most of the code yourself, you

usually had a pretty good feel for the system.



You knew where things lived.



You knew why certain decisions were made.



You could jump into a bug without feeling like you were reading

someone else's project for the first time.



That is changing fast.



Now it is very easy to produce code.



But it is much harder to stay close enough to that code that you

can still defend it, review it properly, and fix it when

something breaks later.



And I think that is where a lot of people are fooling themselves.



They see a huge jump in output and assume that means they are

getting more productive.



Sometimes that is true.



But sometimes they are just creating more code than they can

actually understand.



And that is not leverage.



This is also why I think beginners are in such a weird spot right

now.



They can generate a lot of code very quickly.



But if they cannot explain why the model made a certain choice,

what the tradeoff was, or what should happen when the output is

wrong, then they are not really building engineering judgment.



They are just moving faster on top of shallow understanding.



And to be fair, this is not only a beginner problem.
